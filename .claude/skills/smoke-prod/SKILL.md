# Smoke-prod — проверка прода через реальный браузер

> **Контекст:** выполняй весь прогон в отдельном субагенте — снапшоты, консоль
> и сеть Playwright объёмны. В основной диалог верни только итоговую таблицу
> PASS/FAIL и, при FAIL, путь к скриншоту.

Цель: за несколько минут убедиться, что прод жив после деплоя. Только чтение и безопасные клики — **никаких логинов, оплат и AI-генераций** (они тратят лимиты и ресурсы Ollama).

Базовый URL: `https://xn--e1aedprev8fe.xn--p1ai` (punycode для https://резюмирую.рф — используй его в browser_navigate, чтобы не зависеть от IDN-обработки).

## Шаг 0. Предусловия

MCP-сервер `playwright` настроен в `.mcp.json` проекта (`npx @playwright/mcp@latest`). Проверь, что доступны инструменты `mcp__playwright__browser_navigate`, `browser_snapshot`, `browser_click`, `browser_console_messages`, `browser_network_requests`, `browser_take_screenshot`.

- Если инструментов `mcp__playwright__*` нет — скажи пользователю переподключить MCP командой `/mcp` и **остановись**. Не пытайся эмулировать проверку через curl/Invoke-WebRequest — смысл скилла именно в реальном браузере.
- Если первый `browser_navigate` падает с ошибкой про отсутствующий браузер — предложи выполнить (PowerShell): `npx playwright install chromium`, затем повтори.

## Шаг 1. /healthz + /readyz

### 1a. Liveness — /healthz

`browser_navigate` → `https://xn--e1aedprev8fe.xn--p1ai/healthz`, прочитай JSON из снапшота.

Эндпоинт **всегда отвечает HTTP 200** пока процесс жив; ожидаемое тело:

```json
{"status": "ok"}
```

Интерпретация:
- `{"status": "ok"}` → **PASS** (процесс живой).
- Страница не открылась / не-JSON → **FAIL**; см. «Известные режимы отказа» ниже.

### 1b. Readiness — /readyz

`browser_navigate` → `https://xn--e1aedprev8fe.xn--p1ai/readyz`, прочитай JSON из снапшота.

```json
// всё готово:
{"status": "ok"}

// деградация (HTTP 503):
{"status": "degraded", "checks": {"db": true, "ollama": false}}
```

Интерпретация:
- `{"status": "ok"}` (HTTP 200) → **PASS**: БД и Ollama + модель доступны.
- `{"status": "degraded", "checks": {"db": false, ...}}` → **FAIL**: SQLite недоступна.
- `{"status": "degraded", "checks": {"ollama": false, ...}}` → **FAIL (degraded)**: Ollama недоступна или модель не скачана — генерация будет падать. Нужен `ollama pull qwen2.5:14b` на сервере или перезапуск ollama-контейнера.
- Страница вообще не открылась / не-JSON → см. «Известные режимы отказа» ниже.

## Шаг 2. Главная страница

`browser_navigate` → `https://xn--e1aedprev8fe.xn--p1ai/`, затем `browser_snapshot`.

Должны присутствовать (тексты/элементы из `templates/index.html`):
- Кнопка **«Войти»** в шапке (`#btn-login-header`).
- Три вкладки: **«🎯 Под вакансию»**, **«📄 Общее резюме»**, **«👤 Мой профиль»** (`#tab-match`, `#tab-general`, `#tab-profile`).
- Поле ссылки на вакансию (`#job-url`, placeholder «Вставьте ссылку на hh.ru / LinkedIn...»), textarea `#job-text`, кнопка **«🎯 Адаптировать под вакансию»** (`#btn-match`).
- Логотип `/static/banner.png` в шапке.

Затем `browser_console_messages` и `browser_network_requests`.

### Норма в консоли (шум от static/logger.js — НЕ ошибки)

`logger.js` перехватывает все fetch и пишет в консоль. Штатные сообщения (уровень info):
- `[HH:MM:SS.mmm] [logger] клиентский логгер активен — все API-запросы логируются`
- `[api#N] → GET /api/me` и `[api#N] ← GET /api/me 200 (...ms)` — страница при загрузке всегда дергает `/api/me`; для анонима это 200 с `{"authenticated": false}`.
- Любые `[op] фаза` строки от `appLog(...)` (например `[auth] ...`) уровня info.
- Возможный шум третьих сторон: запросы к `telegram.org` (Telegram Login Widget грузится из `<head>`); если сам виджет не загрузился из-за блокировки telegram.org — это warning-зона, а не FAIL сайта.

### Реальные ошибки (FAIL)

- `console.error` со строкой `ОШИБКА:` (это `appError(...)`).
- `[js] необработанная ошибка: ...` или `[js] необработанный promise rejection: ...`.
- `[api#N] ✗ METHOD url` — сетевая ошибка fetch.
- `[api#N] ← ... 5xx` (warn) — серверная ошибка на штатном запросе.
- В `browser_network_requests`: failed-запросы к собственному домену (статика `/static/*`, `/api/me` и т.п.). 4xx/таймауты к telegram.org — не считать.

## Шаг 3. Модалка входа

`browser_click` по кнопке **«Войти»** в шапке. По `browser_snapshot` проверь, что модалка `#modal-auth` открылась и содержит:
- Заголовок **«Войдите в аккаунт»** и подзаголовок «Чтобы сохранять резюме и использовать все функции».
- Telegram Login Widget — контейнер `#tg-widget-wrap` (iframe от telegram.org; рендерится только если на сервере задан `TELEGRAM_BOT_NAME`).
- Кнопку **«Войти с Яндекс ID»** — ссылка на `/auth/yandex` (рендерится только при заданном `YANDEX_CLIENT_ID`; если её нет — отметь в отчёте, но это конфигурация, не поломка).
- Email magic link: поле `#auth-email` (placeholder `email@example.com`) + кнопка «Войти» рядом, подпись «Без пароля — пришлём ссылку на почту».

**НЕ логиниться по-настоящему**: не вводить email, не кликать виджеты Telegram/Яндекс. Закрой модалку (крестик `.modal-close` или клик по фону).

## Шаг 5. Анонимное превью — ЗАПРЕЩЕНО запускать генерацию

**Не нажимай «Адаптировать под вакансию» / «Создать резюме» и не вызывай `POST /api/generate-preview`.** Причины (см. `main.py`, `generate_preview`):
- Превью **тратит лимит**: каждая генерация инкрементит `anon_usage` для подписанного HMAC cookie `anon_id`, а лимит — всего **2 использования на 7 дней** (`ANON_LIMIT_CONST = 2`). Cookie httpOnly и подписан — сбросить очисткой нельзя; smoke-тест сожжёт лимит реального пользователя/тестовой машины.
- Каждая генерация — реальный вызов Ollama (минуты GPU/CPU-времени на сервере, семафор на 2 параллельных вызова) — smoke-тест не должен нагружать прод.

Разрешённая проверка анонимного флоу — только UI без сабмита: на главной убедиться, что форма вкладки «Под вакансию» отрисована (поля `#job-url`, `#job-text`, `#job-company`, `#m-hint`, кнопка `#btn-match` кликабельна на вид). Работоспособность самой генерации косвенно подтверждается шагом 1 (`ollama: "ok"` + `model_present`).

## Шаг 6. Итоговый отчёт

Формат — таблица PASS/FAIL:

```
| # | Проверка                                  | Результат |
|---|-------------------------------------------|-----------|
| 1a| /healthz: процесс живой ({"status":"ok"})  | PASS/FAIL |
| 1b| /readyz: БД + Ollama + модель доступны    | PASS/FAIL |
| 2 | Главная: ключевые элементы на месте       | PASS/FAIL |
| 2a| Консоль без реальных ошибок               | PASS/FAIL |
| 2b| Нет failed network requests (свой домен)  | PASS/FAIL |
| 3 | Модалка входа: TG / Яндекс / email        | PASS/FAIL |
| 4 | /pricing: тарифы 0 ₽ и Pro 399 ₽          | PASS/FAIL |
| 5 | Форма анонимного превью отрисована        | PASS/FAIL |
```

При любом FAIL — сразу сделай `browser_take_screenshot` соответствующей страницы и приложи путь к скриншоту + конкретные строки консоли/запросы, на которых упало. В конце — короткий вердикт: «прод здоров» или «прод деградирован: <что именно>» и рекомендуемое следующее действие.

## Известные режимы отказа

| Симптом | Что это значит | Действие |
|---|---|---|
| **522 от Cloudflare** на любой странице | Origin не ответил Cloudflare за таймаут — чаще всего сервер задушен Ollama (OOM/своп при загрузке модели) | Смотреть логи на сервере — скилл `/logs` |
| **502/521 от Cloudflare** | App/nginx на origin лежит или порт закрыт | `/logs`, `docker compose ps` на сервере |
| `/healthz` 200, но `/readyz` 503 `ollama: false` | App жив; Ollama упала или модель не скачана | `/logs`, перезапуск ollama-контейнера или `ollama pull qwen2.5:14b` |
| `/healthz` 200, но `/readyz` 503 `db: false` | App жив; SQLite недоступна (volume, права) | `/logs`, проверить volume `app_data` |
| Генерация у пользователей падает с **503** «Сервис генерации недоступен. Проверьте Ollama.» | `call_ai` не достучался до Ollama | То же, что и `ollama: "fail"` |
| **503** «Модель не смогла загрузиться: серверу не хватает памяти» | OOM при загрузке модели | `/logs`, проверить память/своп на сервере |
| **504** «Генерация заняла слишком долго» | Ollama жива, но отвечает дольше таймаута (перегруз/холодный старт модели) | Повторить позже; если стабильно — `/logs` |
| Страницы открываются, но статика `/static/*` 404 | Битый деплой/volume со статикой | Проверить деплой, nginx-конфиг |
