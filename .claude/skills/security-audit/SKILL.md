# Security-audit — аудит и починка безопасности приложения

> **Контекст:** разведку (шаги 1–2: grep и чтение участков `main.py`) выполняй
> в субагенте и верни список находок. Починку — правки `main.py` — делай в
> основном контексте сам, чтобы diff был виден для ревью и коммита.

Полный проход по безопасности всего приложения, а не только diff'а (для diff есть встроенный `/security-review`). Заточен под стек: FastAPI + SQLite + Telegram/Email/Яндекс auth + ЮKassa + MCP. По умолчанию **находит и чинит**: безопасные фиксы применяет к рабочему дереву, рискованные — только описывает с обоснованием. Ничего не коммитит сам — пользователь ревьюит diff (хотя по умолчанию коммит+пуш разрешён, для правок безопасности всё равно дай человеку посмотреть diff перед коммитом).

## Шаг 0. Режим запуска

- `/security-audit` — полный аудит всего приложения (дефолт).
- `/security-audit --diff` — только изменения текущей ветки (быстро, перед мёржем).
- `/security-audit --report` — только отчёт по severity, без правок.

## Шаг 1. Быстрая триажная разведка

Прогони набор grep'ов по `main.py` и шаблонам, чтобы быстро локализовать подозрительные места (это разведка, а не вердикт — каждое попадание потом читать в контексте):

- **SQL-инъекции**: f-string/`.format`/конкатенация внутри `execute(` — `execute\(\s*f["']` , `execute\(.*%`, `execute\(.*\+`.
- **Секреты в коде**: `SECRET_KEY`, `TOKEN`, `SECRET`, `PASSWORD`, `api_key` с присвоением строкового литерала; проверь, что `.env` в `.gitignore`.
- **Логирование чувствительного**: `log`/`print` рядом с `token`, `password`, `secret`, `Authorization`, `session_id`.
- **Небезопасный рендер**: в шаблонах `| safe`, `{% autoescape false %}`, `innerHTML`, `Markup(`.
- **Слабые сравнения**: сравнение подписей/токенов через `==` вместо `hmac.compare_digest`.
- **CORS / cookie**: `allow_origins=["*"]`, отсутствие `Secure`/`HttpOnly`/`SameSite` у `set_cookie`.

## Шаг 2. Аудит по болевым точкам проекта

Прочитай соответствующие участки `main.py` и проверь по чеклисту. Это места, где баг безопасности дороже всего.

### Авторизация
- **Telegram Login Widget**: HMAC-SHA256 подпись проверяется через `hmac.compare_digest` (не `==`); `auth_date` отвергается, если старше 1 часа; `id`/`hash` обязательны.
- **Email magic link**: токен одноразовый (инвалидируется после использования), TTL 15 мин соблюдается, токен — криптослучайный UUID, нет таймингов-утечек.
- **Яндекс OAuth**: проверка `state` (CSRF), валидация `client_id`/секрета, обмен кода на сервере.
- **Сессии**: cookie `session_id` — `HttpOnly`, `Secure` (в проде по `APP_URL`), `SameSite=Lax/Strict`; срок 30 дней осмыслен; при логауте сессия реально удаляется из БД.
- **Анонимный cookie `anon_id`**: HMAC-подпись проверяется, подделать/переиспользовать нельзя для обхода `free_left`.

### Платежи (ЮKassa)
- Вебхук `/api/pay/webhook` **перепроверяет платёж напрямую через API ЮKassa**, а не доверяет телу запроса.
- **Идемпотентность**: повторный вебхук по тому же платежу не выдаёт Pro/пачку дважды.
- Сумма и статус платежа сверяются с ожидаемыми перед выдачей; нельзя выдать Pro за чужой/недоплаченный платёж.
- Секрет ЮKassa не логируется и не утекает в ответы.

### SQL и доступ к данным (IDOR)
- Все запросы параметризованы (`?`-плейсхолдеры), нет f-string/конкатенации в SQL.
- Любой доступ к резюме/версиям фильтруется по `user_id` текущей сессии — нельзя прочитать/изменить/удалить чужой ресурс по `resume_id`.
- Эндпоинты редактора и экспорта проверяют владельца.

### MCP и API-токены
- `get_profile` / `adapt_resume` требуют валидный `Authorization: Bearer <token>`, сверяемый с таблицей `api_tokens` (через `compare_digest`).
- `POST /api/mcp-token` выдаёт один активный токен на пользователя; старый инвалидируется.
- Токен не печатается в логах; при невалидном токене — 401, без утечки деталей.

### Лимиты и расход
- `_deduct()` / `_refund()` атомарны; при ошибке AI лимит корректно возвращается (нет «съедания» без результата и нет бесплатной генерации при гонке).
- Нельзя уйти в минус или обойти `free_left`/`paid_left`/`is_pro` через параллельные запросы.
- `FREE_RESUMES`/`FREE_RESUMES=5` лимит хранимых резюме реально enforced.

### Инфраструктура запроса
- `SECRET_KEY` обязателен в проде (приложение не стартует/не работает в небезопасном режиме без него).
- CORS привязан к `APP_URL`, не `*` с `allow_credentials`.
- Rate limiting (slowapi) покрывает дорогие/чувствительные эндпоинты (вход, генерация, выдача токена).
- LLM-промпты: пользовательский ввод не ломает структуру промпта (prompt injection не приводит к выдаче чужих данных/смене инструкций с последствиями).
- Заголовки безопасности (CSP/`X-Content-Type-Options`/`X-Frame-Options`) — отметить отсутствие как low-severity рекомендацию.

## Шаг 3. Классификация находок

Каждую находку оформи: **severity** (Critical / High / Medium / Low), где (`файл:строка`), в чём риск, как эксплуатируется, как чинить.

- **Critical/High** — обход auth, IDOR, двойная выдача Pro, SQL-инъекция, утечка секретов.
- **Medium** — слабые флаги cookie, отсутствие идемпотентности без явной двойной выдачи, недостающий rate limit на чувствительном эндпоинте.
- **Low** — отсутствие security-заголовков, мелкие hardening-улучшения.

## Шаг 4. Починка

- **Применяй сразу** (если не `--report`): параметризация SQL, `compare_digest` вместо `==`, добавление `user_id`-фильтра в запрос, флаги cookie, удаление логирования секретов, добавление проверки владельца.
- **Только описывай, не применяй молча**: изменения в логике платежей, схеме БД, потоке авторизации, всё что меняет поведение для пользователей или требует миграции/нового env. Покажи предлагаемый diff и риск, дай решить.
- После правок — короткий прогон проверки, что приложение хотя бы импортируется/синтаксически валидно (`python -c "import ast; ast.parse(open('main.py',encoding='utf-8').read())"` — на дев-машине Python может быть только в Docker/uv, см. memory).

## Шаг 5. Итог

Сводка: сколько найдено по severity, что починено в рабочем дереве, что осталось на решение пользователя (с конкретными рекомендациями). Покажи `git diff --stat`. Перед коммитом дай пользователю просмотреть diff правок безопасности.

## Когда НЕ использовать

- Нужен аудит только своих изменений перед мёржем — быстрее встроенный `/security-review` или `/security-audit --diff`.
- Проблема не про безопасность (баг логики, падение прода) — это `/logs` или обычная отладка.
- Нужен пентест живого прода — этот скилл про код, а не про работающий сервер.
